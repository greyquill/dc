The platform

Four layers — a stable core, customized on top

Greyquill is an enterprise trust & governance platform. GQ Data is the foundation everything else depends on, so it's built first as a standalone product. The other three layers talk to it only through documented contracts, so they build in parallel.

GQ Data
Enterprise data-truth layer — ingest, resolve, classify, lineage, quality, governed API.
◀ current focus
GQ Govern
Policies, controls, approvals, evidence + the Diagnose role. Talks to GQ Data via contracts only.
GQ Agents
Governed workflow execution.
GQStudio
Business-specific customization layer (low-code + optional code).
Why this matters for direction: building GQ Data alone — with contracts as the boundary — is what lets the rest of the platform proceed without coupling. We're working on the right layer first.

The key design decision

Domain-agnostic core + verticals as loadable data

This is the answer to "make it reusable for any use case." The engine knows nothing about insurance. A vertical is a pack — pure data declaring entity types, match keys, survivorship & quality rules. Adding BFSI or Healthcare = a new pack.yaml, zero service-code changes.

🔒 Stable core built

Generic primitives + strategy interfaces, registered by id. No entity ever named here.

bfsi/ → Phase 2 (same mechanism)
Litmus test, enforced: if code names PolicyHolder, it belongs in a pack, not the core. LoadPack fails fast if a pack references a strategy that isn't registered.

Service detail

The three Phase-1 services

All three compile today. The generic engine logic (matchers, survivorship, CSV parsing, pack loading) is real Go; the I/O edges (HTTP bodies, Kafka, databases, ML) are wired as stubs with explicit TODOs mapped to backlog stories.

gq-ingest-api compiles

Gin REST gateway. Endpoints exist and validate payloads; persistence + Kafka emit are TODO.

real: routing, request binding

gq-ingest-worker compiles

Connector engine. CSV connector parses files into RawRecords for any entity type.

real: CSV→RawRecord stream
stub: Kafka consume, Mongo store, SAP connector

gq-resolve compiles

The question you asked

Are we heading the right way?

Mapping each strategic intent to concrete evidence in the codebase today.

✓
Reusable across any use case, not insurance-bolted

Core has zero entity names; insurance lives entirely in pack.yaml; loader validates packs. A new vertical is a data file. Directly addresses your ask.

✓
Build GQ Data first, others in parallel

Only GQ Data is scaffolded; the contract boundary (doc 03 / gq-pb) is where Govern plugs in later.

✓
Locked stack, actually compiles

Go + Python, Gin, buf, Kafka/Neo4j/PG planned per doc 05 — green build on real Linux, deps locked in go.sum.

✓
Right first vertical for the thesis

Insurance Claims is Resolve-heavy + audit-heavy — it stresses the hardest component and is the demo centerpiece.

○
End-to-end MVP working

Not yet — this is a scaffold. Real ingest→resolve→golden→audit is the E1–E8 work ahead. Expected at this stage.

○
Proven on a second vertical (BFSI)

The mechanism is ready; BFSI pack is Phase 2. That's when reusability is truly demonstrated, not just designed.

Bottom line. The foundation and the reusability design are in place and verified to compile. The direction is right; what remains is implementing the stubbed logic story-by-story (E0→E8) and proving it on the demo dataset.
